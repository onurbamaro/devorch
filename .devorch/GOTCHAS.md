# Gotchas

- **`git()` wrapper trims stdout** (`scripts/merge-worktree.ts:43-53`) — Callers needing byte-exact output (e.g. `removeIdenticalUntracked` comparing untracked file bytes to `git show <branch>:<path>`) must bypass the wrapper with raw `Bun.spawnSync`; the wrapper's `.trim()` silently alters trailing whitespace and newlines.
- **`parseTasks` exemplars/non-goals regex misses bullet-prefixed lines** (`scripts/lib/slice-builder.ts:98,103`) — The regex `^\s*\*\*Exemplars\*\*:` (and analogous Non-goals) does not match `- **Exemplars**:`, the canonical bullet form used in PLAN-FORMAT.md. Result: `exemplarsByTask` and `nonGoalsByTask` silently return empty for valid plans, so the corresponding sections never reach builder prompts. When debugging missing exemplars/non-goals in a builder prompt, check the regex first before assuming the plan is wrong.
- **Wave file-overlap detection only sees backtick-quoted paths** (`scripts/validate-plan.ts:369-371`) — The fileRefs regex matches only `` `path/to/file.ts` `` style references (backticked, with `/` or extension). A task that mentions a file in plain prose without backticks is invisible to wave conflict detection. Two tasks both touching `src/index.ts` mentioned only in narrative will pass validation and collide at build time. When authoring a plan, always backtick file paths inside task sections.
- **Two file-ref regexes coexist in validate-plan.ts** (`scripts/validate-plan.ts:375 vs 403`) — The strict regex at ~line 375 (warn rule for undeclared body paths) excludes whitespace and `<placeholder>` syntax via `[^`\s<>]+\.(ext)` and a fixed extension allow-list; the lenient regex at ~line 403 (wave-overlap detection) accepts any backtick content with a `/` or short extension. They are intentionally different: the strict form avoids false positives on shell command tokens and template literals; the lenient form captures more for collision detection. When editing either, mirror the change to the other only if the divergence is no longer intended.
- **planDeclaredPaths must union `<relevant-files>` AND `<new-files>`** (`scripts/validate-plan.ts:31-37`) — The undeclared-body-paths warn rule compares each task body's path mentions against `planDeclaredPaths`. That set must include entries from BOTH `<relevant-files>` and `<new-files>`; a naïve implementation that reads only `<relevant-files>` will fire false-positive warnings on every task that touches a legitimately new file declared in `<new-files>`.
